我之前写过一篇博文介绍我写博客的工作流。博客的原始文件都是 Markdown 格式，为了将它们编译成 HTML 以在网页阅读，我还专门开发了一款基于 Nodejs 的编译工具 qingya。但是那篇文章并没有触及编译之后的部署问题。

事实上，我的部署方案就是手动部署，即把编译后的静态博客文件手动上传到 Github，然后 Github Pages 会自动完成托管。但是，每次写作之后，都要手动编译，再手动部署，这不仅无法让人专心于写作，而且每次都要做这些重复性的工作很令人心烦，甚至会打击写作积极性。

像编译、部署这种重复度极高的工作，一定是可以交给计算机来做的，我的理念也一向是尽可能地压榨机器，解放自己。而 Github Actions 就是实现这件事的天然圣体。

# 什么是 Github Actions

> GitHub Actions 是 GitHub 推出的一项强大的自动化工具，它允许用户在 GitHub 仓库中创建、编辑和运行自动化脚本，这些脚本被称为工作流程（workflows）。这些工作流程可以响应 GitHub 上的各种事件，例如代码被推送、issue 被创建、pull request 被打开或定期调度任务等。
> 
> GitHub Actions 的工作流程（workflow）由 YAML 文件定义，这些文件放在仓库的 `.github/workflows` 目录下。每个工作流程文件描述了一系列的作业（jobs）和步骤（steps），定义了运行环境、触发条件、要执行的任务等。
